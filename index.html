项目分层#

原则#

项目分层的原则是便于代码的复用和维护，保证低耦合和高内聚的思想。即低层级模块不依赖于高等级的模块，高等级模块的更改不会影响到低层级模块。

层级目录#

Project structure

踩坑实录#

  1. annotationProcessor 编译时注解（ARouter/ButterKnife）
    高层组件需要依赖低层组件的第三方时，如需要使用编译时注解annotationProcessor，则必须放在使用的组件的build.gradle中声明，在低层声明高层并不能直接引用。ButterKnife会出现View绑定不成功，ARouter会出现找不到Path。
  2. 第三方库 implementation/api的使用
    如果在低层组件中需要把第三方库暴露给高层组件，则用api来引用第三方，如果只在本级Module使用，则只需要使用implementation，能有效的降低编译时间。
  3. Module中ButterKnife的使用注意
    ButterKnife在Module中使用时，需要把R.id.xx改为R2.id.xx，Module下ButterKnife会生成一个R2文件来标记资源id，但是该资源id不是final类型，在条件判断时不可使用switch，需改为if&else来进行条件判断。
  4. App的BuildConfig.DEBUG设置boolean值后，组件的BuildConfig.DEBUG不生效
    BuildConfig.java 是编译时自动生成的，并且每个 Module 都会生成一份，以该 Module 的 packageName 为 BuildConfig.java 的 packageName。所以如果你的应用有多个 Module 就会有多个 BuildConfig.java 生成，而上面的 Lib Module import 的是自己的 BuildConfig.java，编译时被依赖的 Module 默认会提供 Release 版给其他 Module 或工程使用，这就导致该 BuildConfig.DEBUG 会始终为 false。
  5. ARouter 问题汇总
    • Library（模块）的名字前缀不能相同
      例如：module_one、module_tow。Arouter处理时会初始化失败。

新建Module注意事项#

如果Module需要在Application做初始化操作#

步骤：

  1. 建立[ModuleName]InitLogic类，继承BaseAppLogic
  2. 在AppAppcation的initLogic方法中注册组件的BaseAppLogic。
  3. 重写BaseAppLogic的父类方法，方法名就等同于Application的生命周期的方法名。

如果Module需要单独打包调试#

步骤：

  1. 在需要单独编译的Module src目录下建立debug目录(与mian的目录结构一致)，新建应用启动的Activity

  2. 在debug的AndroidManifest配置启动的Activity

  3. 修改basic_config.gradle下新增开关参数is[ModuleName]Run，代表该组件单独编译

  4. 在application（app）下的build.gradle根据开关参数is[ModuleName]Run判断是否依赖组件。

  5. 在Module下的build.gradle配置开关参数is[ModuleName]Run

  6. 在Module下的build.gradle配置SourceSet，根据开关参数is[ModuleName]Run选择不同的项目目录

历史尝试方案#

  1. RecyclerView方案
    方案描述：
    将三大模块作为ItemView来显示
    方式失败原因：
    WebView作为列表的一项触发了Rv的回收机制，在滑动到下方时会把上边的WebView视图回收掉。导致新闻模块高度重绘为0
    出现现象：
    新闻模块被回收，不显示

    ​

  2. ViewGroup嵌套WebView+RecyclerView方案
    方案描述：
    ViewGroup控制页面滑动，WebView负责加载新闻模块，Rv负责推荐文章和评论。
    方式失败原因：
    ViewGroup负责WebView&Rv的中间滑动层，需要处理滑动冲突、惯性滑动处理、临界点高度等条件，过于复杂，容错性较差。此方案思路可行，部分app也是采用此方案，采用此方案需要大量时间调试，维护成本较高。
    出现现象：
    滑动卡顿、模块临界点出现重叠、反向滚动、惯性滑动不流畅（今日头条是此方案，偶现模块临界点出现重叠）

    ​

  3. NestedScrollView嵌套WebView+RecyclerView方案
    方案描述：
    由于方案2的大量滑动逻辑和高度逻辑计算比较复杂，于是想到用NestedScrollView来负责滑动和高度的逻辑计算。
    方式失败原因：
    ScrollView嵌套可滑动组件会出现回收机制失效（当时没想到评论会很多的情况）
    出现现象：
    评论模块为无限条目，不能回收导致内存浪费，加载评论越多内存使用越多越卡顿。

    ​

  4. 最终可行性方案
    采用方案1，重写WebView，Rv对WebView设置不回收。done
    滑动流畅、View可以回收、WebView高度正常

1.0 需求描述#

需求：用户在通过外部的链接（用户分享的安装apk的链接）安装apk之后，需要在apk安装之后拿到用户信息（邀请平台/邀请用户id/邀请码…）以便进行归因或者其他操作。

分析：观察需求可以分析出几个流程

  1. 如何获取用户信息
    在点击下载的url中加入需要写入的参数（eg：www.downloadapk.com?user_id=123&name=tom）
  1. 如何将内容写入apk
    这个问题才是关键，随便就把内容写进去当然是不可能的，google的签名机制就不同意。所以我们需要保证apk不会被损坏的情况下写入内容，而签名目前又分为两种：v1和v2，不同的签名机制显然对应不同的写入和读取方法。具体说明见 [2. 读取和写入]
  1. 如何在安装后读取内容
    见 [2. 读取和写入]

2.0 写入与读取#

原理#

Apk的签名分为v1和v2签名，v1签名是将Apk的所有文件进行签名验证，而v2则是对整个Apk的验证（上图的4个区块的位置和大小等）

2.1 Signature Scheme v1 写入内容#

v1签名只会对Apk的每个文件进行签名，而apk也是一个zip文件（zip文件分为几个部分，其中zip就有一块区域可写入长度是无限制的，那就是comment区域），所以可以在comment区域写入内容。

在APK Signature Scheme v2新的签名方案中，签名信息会被保存在Apk Signing Block（签名信息保存的id为0x7109871a），除了该区域外对其他三个区域做任何操作都不能通过签名验证。所以写入内容就需要在这个区域来搞事情。

2.2.1 内容写入#

v2的写入需要在客户端在打包完成之后，在Apk Signing Block区域插入长度为1024字节的tab（ASCII=11）的占位符得到【空白占位的apk】并得到插入的【占位符起始位置】，上传到服务器作为母apk。用户通过不同的链接（携带需要写入的内容）点击下载，服务器拿到链接的参数和占位符起始位置进行动态写入，然后下载给用户【写入内容的apk】

客户端：
用payload_writer-1.1.6.jar对已签名的apk插入空白占位符，得到【空白占位的apk】和【占位符起始位置】（控制台会输出起始位置）
命令：java -jar payload_writer-1.1.6.jar [apkpath]
将apk和起始位置上传到后台
